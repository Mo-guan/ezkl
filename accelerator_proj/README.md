# ZKML 加速器：零知识机器学习推理的软硬件协同设计

> **目标会议**: ISCA / MICRO / HPCA / ASPLOS  
> **项目状态**: 研究与构思阶段

---

## 1. 项目概述

### 1.1 研究动机

零知识机器学习（ZKML）实现了可验证的 ML 推理 —— 证明者可以证明某个 ML 模型在给定输入上被正确执行，同时不泄露私有数据。**ezkl** 是一个领先的开源框架，能够将 ONNX 模型编译为基于 Halo2 的 zk-SNARK 电路并生成证明。

然而，ZKML 推理比标准推理**慢数个数量级**：
- **前端（Witness 生成）**：模型执行时必须记录所有算术运算的*每一个中间值*（trace/witness）。对于 ResNet-18，这意味着数百万次域元素乘法、加法和查找表操作。
- **后端（ZKP 生成）**：Witness 被多项式承诺方案消费，需要 NTT（数论变换）、MSM（多标量乘法）、多项式求值和哈希计算 —— 所有运算都在大素数域上进行。

**内存占用是灾难性的**：由于*所有*中间值必须作为 witness 保留，内存需求可达标准推理的 100–1000 倍，因此必须使用**模型切分**来适应硬件约束。

### 1.2 拟议架构

加速器在 FPGA 上采用**统一前端 / 承诺后端**协同设计：

| 组件 | 位置 | 功能 |
|------|------|------|
| 模型级切分 | CPU | 将模型图划分为子电路 |
| 预计算 | CPU | 生成证明密钥、固定多项式 |
| **统一前端** | FPGA | 所有域元素多项式计算：ML 推理 + Trace 记录 + ZKP 多项式约束求值（VPU / 可重构模运算阵列）|
| **承诺后端** | FPGA | NTT 域变换 + MSM 多项式承诺 |
| Multiopen | CPU | 最终证明组装和多开放论证 |

**关键架构洞察**：证明流程天然分解为 **6 轮** "前端 poly 计算 → 后端 NTT + MSM 承诺"。Witness 生成和 ZKP 多项式求值都是同一素数域上的域元素算术——它们可以共享同一硬件阵列。只有 MSM（椭圆曲线点运算）需要专用单元。

**前端关键操作**涵盖 ML 推理和 ZKP 约束：
- 矩阵乘法被分解为**点积**操作，每个点积结果都需要被 trace
- **量化**使用域元素算术执行（而非浮点）
- 位运算处理域元素的**拆分/聚合/量化**，用于范围检查查找表

### 1.3 基线贡献

1. **首个 ZKML 硬件加速器**，面向端到端可验证推理
2. **统一前端架构**，在同一可重构模运算阵列上同时处理 ML witness 生成和 ZKP 多项式约束求值，与专用 NTT+MSM 承诺后端形成流水线
3. **多轮片上 poly→commit 流水线**，利用 Halo2 证明的天然 6 轮结构，片上数据复用消除多项式计算与承诺之间的 DRAM 往返
4. **跨轮次调度与动态资源重平衡**，缓解模型相关的前后端比例变化（实测单模型内各轮次从 1:21.5 到 1.47:1 剧烈波动），包括跨轮次重叠、evaluate_h 分解流水、MSM 功耗门控
5. **端到端片上执行**，结合模型切分，支持 ezkl 的 **polycommit** 证明拼接方案

### 1.4 评估计划

- 在 ezkl/Halo2 上针对 ResNet-18 进行 FPGA 实现
- 与 CPU（ezkl 原生）和 GPU（Icicle 加速的 ezkl）进行对比

## 2. 头脑风暴：新颖研究想法

> 以下想法旨在加强本工作超越"首个 ZKML 加速器"的新颖性。每个想法都分析了**创新理由**和**可行性**。

---

### 想法 1：连续后端证明之间的动态数据复用

**核心洞察**：当模型被切分为 *N* 个分区时，后端顺序生成 *N* 个子证明。目前每个子证明被视为独立任务 —— 所有中间数据在下一次证明开始前被丢弃。但在 ZKML 中，连续子证明**并非独立**：分区 *i* 的输出激活就是分区 *i+1* 的输入激活，产生**重叠的 witness 内容**，这些内容会传播到多项式和承诺计算中。核心问题是：**分区 *i* 后端证明生成过程中产生的动态中间结果，能否在证明分区 *i+1* 时直接复用？**（这不同于复用旋转因子或 SRS 点等静态协议常数，那些复用是显而易见的。）

**描述**：
- **边界 witness 多项式共享**：在 ZKML 模型切分中，分区 *i* 的输出激活恰好是分区 *i+1* 的输入激活。在电路中，这些值作为 witness（advice）列值出现。当后端对分区 *i* 的 witness 进行承诺时，会对这些值计算 NTT 和多项式承诺。如果相同的值在分区 *i+1* 的电路中再次出现为 witness 条目，**对应的多项式系数和部分 NTT 蝶形运算结果可以保留**而非从头重新计算。
- **查找表论证中间数据复用**：ezkl 通过查找表实现非线性运算（ReLU 等）。连续分区通常使用**相同的查找表**且输入范围重叠。后端的查找表论证需要计算排序置换和表项的多项式承诺 —— 这些中间排序表及其 NTT 表示可以在共享相同查找配置的子证明之间**缓存和复用**。
- **排列论证数据流**：Halo2 的排列论证强制执行拷贝约束（如"第 *k* 层输出 = 第 *k+1* 层输入"）。这些约束的证明生成中间多项式积（grand-product 多项式）。对于跨分区边界约束，分区 *i* 的部分 grand-product 计算包含与分区 *i+1* 排列论证直接相关的项。
- **部分多项式求值复用**：在 IPA/KZG 的开放阶段，证明者在随机挑战点处求值已承诺的多项式。如果连续子证明共享多项式内容（来自共享的 witness 数据），**部分 Horner 求值可以复用**，减少冗余的域乘法。
- **证明感知切分器**共同优化切分点以最大化连续证明之间的动态数据重叠：(a) 在最大化共享边界激活的位置切分；(b) 对齐分区电路以复用相同的查找表配置；(c) 与前端内存约束平衡。

**创新理由**：
- 现有 ZKP 加速器（PipeZK、CycloneNTT、GZKP）将每个证明视为独立、无记忆的任务。在通用 ZKP 中，连续证明**结构无关**（如不同交易），因此证明间数据复用不可能实现。
- 带模型切分的 ZKML 是**独特的**：它产生一系列证明，其中连续证明具有**语义相关的 witness 内容**（共享激活、相同查找表、跨分区拷贝约束）。这种结构关系在任何其他 ZKP 工作负载中都不存在。
- 前端数据复用（激活分块、权重复用）在 ML 加速器文献中已被广泛研究。此处的新颖性在于 **ML 层面的数据关系通过 ZKP 编译传播到后端证明中**，在 NTT、MSM、查找表和排列计算中创造了前人未曾识别的复用机会。
- 开辟新研究问题：**ML 模型结构如何影响 ZKP 证明生成的数据流拓扑？**

**可行性**: ★★★☆☆
- 需要深入分析 ezkl 编译器如何将 ML 操作映射到 Halo2 电路列，以及边界激活如何出现在 witness 多项式中 —— 可通过追踪 ezkl 约束生成实现
- 查找表复用较直接：使用相同非线性函数的连续分区共享相同的表
- 部分 NTT 复用取决于 witness 重叠程度 —— 需通过 ResNet-18 切分分析量化
- 排列论证数据复用需要理解 Halo2 在分区边界处的 grand-product 多项式结构
- 实现：离线分析识别可复用数据 + 硬件暂存存储保留证明间中间结果

### 想法 2：流式 Witness-to-Proof 流水线与片上暂存调度

**核心洞察**：当前 ZKML 流程是严格**顺序**的 —— 先生成所有 witness，再进行证明。在内存受限的 FPGA 上，这意味着一个分区的整个 witness 必须存储后才能开始证明。相反，我们可以以**列粒度**将 witness 数据**流式传输**给证明器，并且实现分区间前端/后端的**流水线重叠**。

**描述**：
- 在 Halo2 中，电路组织为列 × 行的矩阵。每列独立地通过多项式承诺进行承诺。
- **分区内流式处理**：前端可以**逐列生成** witness（通过适当的数据流调度），后端可以在*整个 witness 准备好之前*就开始对已完成列进行 NTT + 承诺。
- **跨分区流水线重叠**：当后端正在证明分区 *i* 时，前端可以同时为分区 *i+1* 生成 witness。这种**跨分区流水线重叠**将前端延迟隐藏在后端计算之后，显著缩短多分区模型的端到端证明时间。
- 新型**片上暂存调度器**管理：(a) 分区内前端列生成和后端列承诺之间的双缓冲；(b) 编排跨分区的前端/后端重叠，确保正在生成的 witness（分区 *i+1*）和正在证明的 witness（分区 *i*）之间的内存隔离。

**创新理由**：
- 将 ZKML 流水线从批处理转变为**流式处理**，同时减少延迟和峰值内存。
- 现有 ZKP 加速器均未利用列级承诺并行性 —— 都在等待完整 witness。
- **跨分区流水线重叠**由 ZKML 中的模型切分独特启用 —— 通用 ZKP 中不存在类似机会。
- 为 ZKP 证明硬件引入了新的**流式数据流**范式。

**可行性**: ★★★☆☆
- 需要理解 Halo2 排列论证中的列间依赖关系
- 使用查找表论证的列可能需要多列同时就绪
- 跨分区流水线需要足够的片上内存来同时容纳正在证明的 witness 和正在生成的下一个 witness
- 可能需要分组"列集"而非单独列 —— 仍可提供显著收益

### 想法 3：统一前端 Poly→Commit 流水线与片上数据复用

**核心洞察**：对 ezkl 证明流程的详细 profiling 揭示了一个**重复的多项式计算→承诺模式**。不同于传统划分（前端 = 仅 ML 推理，后端 = 全部 ZKP），我们观察到 witness 生成**和** ZKP 多项式约束求值都是**纯域元素算术**——它们的计算原语相同，只是数据流不同。而承诺操作（椭圆曲线上的 MSM）则根本不同。这驱动了一种**统一前端**处理所有多项式计算（ML witness + ZKP 约束），配合**专用承诺后端**（NTT + MSM），形成多轮片上流水线。

**描述**：

证明流程天然分解为 6 轮 "前端 poly 计算 → 后端 commit"：

| 轮次 | 前端（Poly 计算） | 后端（NTT + MSM 承诺） |
|------|-------------------|----------------------|
| 1 | witness_synthesize + batch_invert | advice_commit (MSM) |
| 2 | lookup compress + count_m | commit_m (MSM) |
| 3 | permutation grand-product poly | commit_z (MSM) + z NTT |
| 4 | lookup log-derivatives + phi 构造 | commit_phi (MSM) + m/phi NTT |
| 5 | **evaluate_h**（custom gates + perm + lookup + shuffle 约束） | advice NTT（前置） + vanishing commit（后置） |
| 6 | 多项式求值 + SHPLONK poly | SHPLONK commit (MSM) |

- **统一前端计算阵列**（VPU 或可重构模运算阵列）执行：
  - ML 推理的域元素 MAC（witness 生成）—— 轮次 1
  - ZKP 多项式约束求值的域元素运算 —— 轮次 2–6
  - 相同硬件，不同数据流调度
- **专用承诺后端**处理：
  - NTT 蝶形引擎用于 Lagrange↔Coeff↔Extended 域变换
  - MSM 引擎用于多项式承诺（椭圆曲线点运算）
- **片上数据复用**：前端计算的多项式值直接流式传输到后端进行 NTT + MSM 承诺，*无需 DRAM 往返*。每轮的中间数据保留在片上。

**流水线空泡问题与缓解策略**（基于 profiling 量化）：

前端/后端计算比例在不同轮次间**剧烈波动**，产生流水线空泡：

| 轮次 | 前端 (ms) | 后端 (ms) | F:B 比例 | 主导空泡 |
|------|----------:|--------:|--------:|:---------|
| 1. Witness→Advice | 726 | 7,814 | 1:10.8 | 前端空闲 |
| 2. Lookup Prep→m | 4,359 | 7,671 | 1:1.8 | 平衡 |
| 3. Permute→z | 3,161 | 52,974 | 1:16.8 | 前端空闲 |
| 4. Lookup GS→phi | 3,650 | 78,463 | 1:21.5 | 前端空闲 |
| 5. evaluate_h→vanish | 125,144 | 85,179 | **1.47:1** | **后端空闲** |
| 6. PolyEval→SHPLONK | 11,048 | 22,133 | 1:2.0 | 前端空闲 |
| **总计** | **148,088** | **266,191** | **1:1.8** | **47.8% 总空泡** |

四种架构策略缓解 47.8% 的空泡开销：

1. **跨轮次重叠**：在第 *N* 轮后端 commit 期间，前端通过双缓冲开始计算第 *N+1* 轮的 poly 工作。例如：第 3 轮 commit（53s）可完全隐藏第 4 轮的全部 poly 工作（3.7s），并部分执行第 5 轮的预计算。

2. **统一计算阵列动态重配置**：前端 poly 求值和 NTT 蝶形都使用模运算，共享阵列可按轮次重新分配单元：
   - 轮次 1–4（后端重）：更多单元分配给 NTT 蝶形执行
   - 轮次 5（前端重）：大部分单元分配给 poly 约束求值

3. **evaluate_h 分解流水**：将主导性的 evaluate_h 阶段（125s，占全部前端工作的 84%）分解为子流水线：custom_gates（17s）→ perm_constraints（10s）→ lookup_constraints（98s），每个子结果在完整 evaluate_h 完成前即可开始流向后端。

4. **MSM 功耗门控**：MSM 单元（椭圆曲线算术，无法与域算术共享）在第 5 轮前端主导的 125s 期间进行功耗门控，减少后端空闲期的能量浪费。

**前后端比例随模型变化**：

1l_sigmoid 上观测到的 36%:64% 前后端比例**不是常数**——它是模型结构的函数：

| 模型特征 | 比例影响 | 机制 |
|---------|:-------:|------|
| 更多非线性类型（Sigmoid+ReLU+Exp） | 前端 ↑ | lookup argument 增多 → `poly_lookup_constraints` 增长 |
| 更大 lookup range | 前端 ↑ | 每个 argument 输入更多 |
| 更深的 conv/einsum 堆叠 | 前端 ↑ | `poly_custom_gates` 增长 |
| 更多 advice 列（更宽模型） | 后端 ↑ | 更多 MSM 承诺 |
| 更大 logrows | 后端 ↑ | 每次 MSM/NTT 更慢 |

*硬件启示*：固定的前后端面积比将导致偏离设计点的模型出现利用率损失。统一计算阵列通过动态重平衡直接解决此问题。

**创新理由**：
- 所有先前 ZKP 加速器将 NTT/MSM 视为唯一加速目标并使用专用单元。**无先前工作识别出** Halo2 证明中固有的多轮 poly→commit 流水线结构，也无人提出同时处理 ML witness 生成和 ZKP 多项式约束求值的统一前端。
- 这种统一**仅由 ZKML 独特启用**：在通用 ZKP 中没有 ML 计算与多项式求值共享。在 ZKML 中，两者都是同一素数域上的域元素算术。
- **流水线空泡分析**和**模型相关比例变化**是全新的架构洞察：它们表明 ZKML 证明不是均匀工作负载，而是动态失衡的流水线，需要可重构硬件——这个问题在先前 ZKP 加速器中不存在。
- 每轮 poly→commit 内的片上数据复用消除了软件实现中占主导的 DRAM 带宽瓶颈。

**可行性**: ★★★★★
- 同一模乘器服务于 witness 生成、多项式求值和 NTT 蝶形——共享无硬件开销
- MSM 是唯一的非共享单元（需要椭圆曲线点运算），但其工作负载特征明确且可预测
- 跨轮次调度是主要挑战，由片上暂存调度器（想法 2）解决
- Profiling 数据已就绪，可验证空泡削减估算
- 详细定量分析见：[`analysis/frontend_backend_ratio_analysis.md`](../analysis/frontend_backend_ratio_analysis.md)

### 想法 4：面向非线性运算的查找表感知 SRAM 架构

**核心洞察**：ezkl 广泛使用**查找表论证**来实现非线性运算（ReLU、sigmoid、除法、比较）。这些查找表支配了电路规模和 witness。专用的 SRAM 架构可以利用其访问模式。

**描述**：
- 在 Halo2 + ezkl 中，非线性函数通过范围检查查找表实现。每个 ReLU 需要将域元素分解为位并执行基于查找的范围检查。
- 查找表在证明期间是*固定*且*只读*的，但具有高空间局部性（连续输入激活值命中相似的表区域）。
- 设计**查找表优化的 SRAM 组**：(a) 热门表项的内容可寻址缓存；(b) 基于激活值范围的预取；(c) 利用域元素冗余的压缩表存储。
- 这也有益于后端，查找表论证需要对表项进行排序和多项式求值。

**创新理由**：
- 查找表论证是 ezkl 电路的主要开销（通常占行数 >60%）—— 优化它们具有超大影响。
- 现有加速器没有 ZKML 特定的存储层次。标准 ZKP 加速器甚至没有查找表感知设计。
- 将 VLSI 存储设计与 ZKP 特定数据结构相连接 —— 对架构会议很有吸引力。

**可行性**: ★★★★☆
- 查找表访问模式可从 ONNX 图静态分析
- 可作为专用 SRAM 控制器集成到现有片上存储旁
- 可通过 ezkl 基准测试上的查找表命中率分析进行衡量

### 想法 5：利用稀疏性的 ZKP 生成 —— 剪枝模型产生更廉价的证明

**核心洞察**：剪枝/稀疏 ML 模型具有零值权重和激活。在标准推理中，稀疏性 → 更少的 MAC。在 ZKML 中，稀疏性 → **零值 witness 单元**仍需完整证明开销。能否在*证明*阶段利用稀疏性？

**描述**：
- 在 Halo2 电路中，零值 witness 单元仍需多项式承诺和约束检查。
- 然而，包含许多零的列具有**稀疏多项式** —— 其 NTT 可以更快计算，MSM 有许多零标量项可以跳过。
- 设计**稀疏性感知证明流水线**：(a) 前端为 witness 列标记稀疏性元数据；(b) 后端使用稀疏 NTT 算法并跳过零标量 MSM 项；(c) **稀疏性感知调度器**动态分配更多计算给稠密列。
- 在模型层面，**面向可证明性的剪枝**训练过程共同优化精度和证明生成成本。

**创新理由**：
- ML 模型压缩与 ZKP 效率的新颖交叉 —— 无先前工作存在。
- 开辟新研究方向："面向可证明性的神经架构搜索"
- 对架构会议有强有力的论点：展示 ML 模型属性如何通过编译传递到硬件效率。

**可行性**: ★★★☆☆
- 稀疏 NTT 算法存在但需适配素数域
- 零标量 MSM 跳过通过条件执行即可实现
- 面向可证明性的剪枝是较长期的研究方向，但加强了论文的愿景

---

### 想法 6：安全 Witness 隔离 —— 面向私有推理的硬件级隐私

**核心洞察**：在私有 ZKML 推理中，输入数据和/或模型权重是秘密的。Witness 包含所有中间值，绝不能泄露。硬件加速器必须提供具有硬件级保证的 **witness 隔离**。

**描述**：
- 前端从私有输入生成 witness。这些 witness 流向后端进行承诺。
- 设计**硬件隔离机制**：(a) witness SRAM 位于物理隔离的存储区域，主机无 DMA 访问权限；(b) 仅最终证明（公开输出）可被主机读取；(c) 前端和后端块之间加密 witness 传输的片上密钥管理。
- 类似于 TEE（可信执行环境）但专为 ZKP 设计，避免通用 TEE 的开销。

**创新理由**：
- 解决软件 ZKP 实现固有的真实安全问题（DRAM 中的 witness 易受冷启动攻击）。
- 现有 ZKP 加速器均未在硬件层面考虑 witness 隐私。
- 对重视性能之外安全贡献的审稿人有吸引力。

**可行性**: ★★★★☆
- 内存隔离在 FPGA/ASIC 设计中是标准做法
- 增加极小的面积开销
- 可通过安全分析 + 性能开销测量进行评估

---

### 想法 7：面向持续/在线 ML 推理的增量证明

**核心洞察**：现实世界的 ZKML 通常用*相同模型*对不同输入进行*重复*推理证明。当前流程每次从头重新证明。**增量证明**架构可缓存模型相关的证明工件。

**描述**：
- 证明密钥、固定列承诺和固定多项式求值在使用同一模型的推理之间*完全相同*。
- 设计片上**证明密钥缓存**，跨推理请求持久化模型特定预计算。
- 对于顺序推理（如时间序列），利用 **witness 增量压缩**：如果连续输入相似，仅利用 Halo2 排列论证结构重新证明变化的 witness 列。

**创新理由**：
- 针对实际部署场景（将 ZKML 推理作为服务），关注摊销成本而非单次延迟。
- 无先前工作在 ZKP 硬件中考虑多推理优化。
- 独特的硬件贡献：专为密码学预计算构建的缓存。

**可行性**: ★★★★☆
- 证明密钥缓存在 FPGA 上使用专用 Block RAM 即可实现
- 基于增量的重新证明需要 Halo2 编译器修改，但概念上可行
- 可衡量的改进：比较冷启动与热启动证明延迟

## 3. 推荐研究重点

基于新颖性影响和可行性，最强的论文叙事结合：

| 优先级 | 想法 | 论文中的角色 |
|--------|------|-------------|
| **核心** | 想法 3（统一 Poly→Commit 流水线）| 架构骨干 —— 关键硬件新颖性。统一前端处理 witness+poly 计算，专用承诺后端，多轮片上流水线与动态资源重平衡。|
| **核心** | 想法 2（流式流水线）| 数据流新颖性 —— 列级流式处理 + 跨分区流水线重叠。与想法 3 的轮间调度互补，提供更细粒度的轮内流式。|
| **核心** | 想法 1（证明间数据复用）| 模型切分场景的调度新颖性 —— 连续子证明间的静态电路上下文复用（同构电路快速切换）。|
| 支撑 | 想法 4（查找表 SRAM）| 存储层次新颖性 —— 查找表感知 SRAM 直接受益于 `poly_lookup_constraints`（evaluate_h 中 66% 的主导前端瓶颈）。|
| 讨论 | 想法 5（稀疏性）| 未来工作 / 愿景部分 |
| 讨论 | 想法 7（增量证明）| 未来工作 / 系统级论述 |

**统一论述**：*"我们提出首个 ZKML 加速器，将 ML witness 生成和 ZKP 多项式求值统一到单一可重构前端计算阵列中，通过多轮片上数据复用架构与专用 NTT+MSM 承诺后端形成流水线，并采用动态资源重平衡来缓解 ZKML 证明中固有的模型相关前后端比例变化——相比 GPU 加速的 ezkl 实现 N 倍加速和 M 倍内存缩减。"*
