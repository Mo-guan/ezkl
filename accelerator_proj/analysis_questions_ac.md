# 关于 A 和 C 的技术可行性分析

## 1. 关于 A：模型切分产生相同电路结构的原理

### 模型与电路结构的映射关系
在 ezkl 中，ONNX 模型到 halo2 电路的映射是**确定性的**。电路结构并不取决于具体的输入数据（Witness），而是取决于模型的拓扑结构。

**核心结论**：电路结构由 `GraphSettings` 决定，包含：
- **`logrows` (k)**：决定多项式度 $2^k$。
- **列配置**：Advice 列、Fixed 列、Instance 列的数量。
- **约束系统**：由算子类型（如 Conv, ReLU）及其排列决定。
- **张量形状**：算子处理的张量维度必须完全一致。

### 如何产生相同的电路结构？
要使连续的子模型产生相同的电路结构（从而共享 Proving Key），必须满足：
1. **同构算子序列**：例如都是 `Conv2d -> ReLU -> Conv2d`。
2. **完全一致的形状**：所有权重和激活值的维度必须相同。
3. **一致的量化参数**：`scale` 和位宽需相同。

> [!TIP]
> **硬件加速机会**：当连续证明同构时，硬件无需重复加载 Proving Key、NTT 旋转因子或 SRS 基点表。

## 2. 关于 C：批量证明（Batch Proving）中的数据流共享

### Halo2 Batch Proving 机制
Halo2 原生支持一次 `create_proof` 传入多个电路实例。

**原理如下**：
- **方案**：`create_proof(..., &[circuit1, circuit2, ...], ...)`。
- **共享转录本**：所有电路共享同一个 Fiat-Shamir 转录本。
- **共享挑战值**：因此 $\theta, \beta, \gamma, y, x$ 对所有电路都是相同的。
- **聚合开放**：SHPLONK 会将所有电路的多项式求值合并为一个 Batch Opening 证明。

### 共享数据流的优势
| 维度 | 独立证明 (N) | 批量证明 (1次) |
| :--- | :--- | :--- |
| **Fiat-Shamir 开销** | N 次哈希挤出 | 1 次哈希挤出 |
| **计算复用** | 无 | 共享挑战值，支持更高效的 MSM 并行化 |
| **证明大小** | $O(N)$ | $O(1)$ 或极低增长 |

### 局限性与代价
1. **同构要求**：Batch Proving 要求所有电路必须共享相同的 `ConstraintSystem`（即同构电路）。
2. **对齐内存**：所有电路必须按最大的 $k$ 值对齐，小电路会导致行浪费。
3. **内存压力**：由于要同时计算多个电路的多项式，内存开销会显著增加。
